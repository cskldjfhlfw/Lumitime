# 概要设计文档: 拾光筑梦 Lumitime

版本：v0.1  
日期：2026-06-12  
适用阶段：技术评审 / 详细设计 / 开发拆分 / 部署规划  
关联文档：PRD、权限矩阵、信息架构、接口设计、数据模型设计、测试计划

## 1. 设计目标

Lumitime 首版系统目标是构建一个个人工作站网页，支持个人内容展示、邀请用户访问、访客留言、工作站服务执行、大屏统计和管理员后台维护。

概要设计重点：

- 明确系统分层和模块职责。
- 明确前台、后台、API、Worker、数据库、文件存储之间的关系。
- 明确日志自动提交服务的异步执行机制。
- 明确权限校验、敏感信息脱敏和审计追溯机制。
- 明确 Docker Compose 部署、更新、备份和回滚策略。

## 2. 架构决策

| 项目 | 决策 |
| --- | --- |
| 技术栈 | 概要设计阶段暂不固定具体技术栈 |
| 架构形态 | 模块化单体，后续可拆分服务 |
| 脚本执行 | 后端创建任务，由独立 Worker 执行脚本 |
| 异步任务 | 首版使用数据库任务表，后续可升级消息队列 |
| 文件存储 | 定义文件存储服务抽象，不绑定具体方案 |
| 权限校验 | 前端控制显示，后端 API 强制校验 |
| 执行日志 | 脚本输出先脱敏再存储，系统只保存脱敏日志 |
| 部署方式 | 推荐 Docker Compose，Web、Worker、DB 分开 |

## 3. 总体架构

```text
用户浏览器
  │
  ├─ 公开页面：主页、留言板、大屏看板

  └─ 管理后台：内容、用户、邀请码、服务、审计、导出
        │
        ▼
Web 应用 / API 服务
  ├─ 认证与会话模块
  ├─ 权限控制模块
  ├─ 内容管理模块
  ├─ 留言模块
  ├─ 工作站服务模块
  ├─ 服务请求模块
  ├─ 看板统计模块
  ├─ 导出模块
  ├─ 审计模块
  └─ 文件存储适配层
        │
        ├───────────────┐

```

## 4. 模块划分

### 4.1 前台模块

职责：

- 展示公开主页。
- 展示随记 / 留言板。
- 展示公开大屏看板。
- 展示邀请用户可访问的脚本、作品、博客。
- 展示工作站服务列表和服务详情。
- 提供日志自动提交服务入口。
- 展示邀请用户自己的提交记录。

关键约束：

- 访客只能访问公开页面。
- 邀请用户访问受限内容前必须登录。
- 前台只能控制显示，不承担最终权限判断。

### 4.2 管理后台模块

职责：

- 管理邀请码。
- 管理用户。
- 管理脚本、作品、博客。
- 管理作品附件下载权限。
- 管理留言。
- 管理工作站服务。
- 查看服务提交记录和完整脱敏执行日志。
- 查看审计记录。
- 导出统计记录和服务提交记录 CSV。

关键约束：

- 仅管理员可访问。
- 所有管理接口必须后端强制校验管理员角色。
- 危险操作需要二次确认。

### 4.3 认证与会话模块

职责：

- Lumitime 站点账号登录。
- 邀请码注册。
- 退出登录。
- 当前用户查询。
- 修改站点密码。

设计说明：

- 概要设计推荐 Cookie Session，但不在本阶段强制绑定。
- 站点密码只保存哈希。
- 学生学习 App 账号密码不属于站点账号体系。

### 4.4 权限控制模块

职责：

- 根据角色判断页面和 API 访问权限。
- 支持访客、邀请用户、管理员三类角色。
- 后端 API 强制权限校验。

权限层次：

```text
前端路由与导航显示控制
  ↓
API 鉴权中间件
  ↓
业务模块权限判断
  ↓
数据查询范围限制
```

示例：

- 邀请用户查询提交记录时，后端必须附加 `lumitime_user_id = current_user.id` 条件。
- 管理员查询提交记录时，可以查询全量记录。
- 访客访问工作站接口时返回未登录或无权限。

### 4.5 内容模块

职责：

- 管理和展示脚本、个人作品、经验心得 / 博客。
- 使用统一内容模型 `content_items`。
- 作品附件通过 `content_attachments` 管理。

关键规则：

- 脚本只提供代码文本，不提供文件下载。
- 邀请用户可以复制脚本代码。
- 作品压缩包是否允许下载由管理员按作品配置。
- 访客不能查看脚本、作品、博客详情。

### 4.6 留言模块

职责：

- 公开展示留言。
- 支持访客和邀请用户提交留言。
- 支持管理员隐藏、恢复、删除留言。

关键规则：

- 留言提交后立即公开。
- 必须提供频率限制、内容长度限制和基础防刷策略。
- 来源 IP 只保存哈希。

### 4.7 工作站服务模块

职责：

- 管理工作站服务列表。
- 展示服务详情和输入 schema。
- 创建服务请求。
- 支持异步执行和状态查询。
- 支持失败重试。

首版服务：

- 日志自动提交。

关键规则：

- 首版所有邀请用户可见全部启用服务。
- 后续预留按单服务授权能力。
- 服务请求创建后立即返回 `service_request_id`。
- 前端通过轮询查询执行状态。

### 4.8 Worker 模块

职责：

- 从数据库任务表中拉取待执行服务请求。
- 调用对应自动化脚本。
- 捕获脚本输出。
- 对脚本输出进行脱敏。
- 更新服务请求状态。
- 写入脱敏执行日志。
- 写入审计记录。

Worker 设计原则：

- 与 Web/API 服务分开进程或容器。
- Worker 不保存原始未脱敏日志。
- Worker 不持久化学生学习 App 密码。
- Worker 执行失败时返回结构化失败分类。

### 4.9 看板统计模块

职责：

- 统计公开大屏指标。
- 生成日度统计快照。
- 提供近 7 天、30 天、90 天趋势数据。
- 支持管理员导出统计快照 CSV。

公开指标：

- 网站用户数。
- 开发者数。
- 访问数。
- 个人作品数。
- 脚本数。
- 博客数。
- 随记数。
- 工作站服务数。

关键规则：

- 大屏只展示聚合数据。
- 不展示用户明细、IP、账号、留言原文、服务提交日志。

### 4.10 文件存储模块

职责：

- 上传作品压缩包或附件。
- 下载被授权附件。
- 删除或替换附件。
- 屏蔽底层存储差异。

设计说明：

- 定义文件存储服务抽象。
- 首版可落地到本地存储或对象存储，但业务接口不直接绑定。
- 不向前端暴露永久公开文件地址。

### 4.11 导出模块

职责：

- 导出统计记录 CSV。
- 导出服务提交记录 CSV。

关键规则：

- 只有管理员可导出。
- 导出文件不得包含学生学习 App 密码、完整账号、Cookie、Token、Authorization。
- 服务记录导出只能包含非敏感字段和脱敏日志。

### 4.12 审计模块

职责：

- 记录关键操作。
- 支持管理员查询。
- 支持按 request_id、用户、时间范围检索。

审计范围：

- 登录。
- 邀请码使用。
- 管理员内容操作。
- 工作站服务请求。
- 服务记录查看。
- 导出操作。
- 敏感管理操作。

## 5. 日志自动提交服务流程

### 5.1 正常执行流程

```text
邀请用户打开日志自动提交服务页
  ↓
输入学生学习 App 账号和密码
  ↓
点击自动提交
  ↓
Web/API 校验用户登录状态和服务可用状态
  ↓
生成 service_request_id
  ↓
保存服务请求记录
  ↓
不保存学生学习 App 密码
  ↓
将任务标记为 pending
  ↓
返回 service_request_id 给前端
  ↓
前端开始轮询服务请求详情
  ↓
Worker 拉取 pending 任务
  ↓
Worker 调用自动化脚本
  ↓
脚本执行并输出日志
  ↓
Worker 对日志脱敏
  ↓
保存脱敏日志
  ↓
更新服务请求状态为 success / failed / timeout
  ↓
前端轮询到终态
  ↓
用户看到结果摘要
```

### 5.2 失败重试流程

```text
用户打开失败记录
  ↓
点击重新提交
  ↓
重新输入学生学习 App 账号和密码
  ↓
创建新的 service_request_id
  ↓
记录 retry_of_service_request_id
  ↓
按正常执行流程重新执行
```

关键规则：

- 不复用历史密码。
- 不从历史记录恢复密码。
- 新请求有新的 `service_request_id`。
- 邀请用户只能重试自己的失败记录。

## 6. 异步任务设计

### 6.1 首版方案

首版使用数据库任务表机制。

核心表：

- `service_requests`
- `service_execution_logs`
- `service_audits`

状态流转：

```text
pending
  ↓
running
  ↓
success / failed / timeout / not_integrated
```

### 6.2 Worker 拉取策略

建议：

- Worker 定时拉取 `pending` 任务。
- 拉取时将任务状态更新为 `running`。
- 同一任务只能被一个 Worker 执行。
- 任务应设置超时时间。
- 任务失败时记录失败分类。

### 6.3 后续升级

当服务量增大或任务并发变高时，可升级为消息队列。

可选方向：

- Redis Queue。
- RabbitMQ。
- 云厂商队列服务。

升级后保持 API 和前端流程不变。

## 7. 敏感信息处理

### 7.1 学生学习 App 凭证处理

| 信息 | 处理方式 |
| --- | --- |
| 学生学习 App 账号 | 请求生命周期内使用；只保存掩码和不可逆哈希 |
| 学生学习 App 密码 | 请求生命周期内使用；不保存、不哈希、不掩码、不导出 |
| Cookie / Token | 不保存；日志中必须脱敏 |
| 原始请求体 | 不保存 |
| 原始执行日志 | 不保存 |

### 7.2 脱敏位置

脱敏必须发生在日志进入存储之前。

```text
脚本输出
  ↓
脱敏处理
  ↓
保存脱敏日志
  ↓
按权限展示或导出
```

### 7.3 脱敏内容

必须脱敏：

- password
- token
- cookie
- authorization
- 完整学生账号
- 请求头
- 可能包含凭证的异常堆栈

## 8. 访问与统计设计

### 8.1 访问记录

- 访问明细短期保存。
- IP 保存哈希。
- User-Agent 保存摘要。
- 定期聚合为日度快照。
- 聚合后可清理访问明细。

### 8.2 日度快照

每天生成一条全站快照。

快照包含：

- 用户数。
- 开发者数。
- 访问数。
- 作品数。
- 脚本数。
- 博客数。
- 留言数。
- 工作站服务数。
- 较前一日变化。

## 9. 部署架构

推荐使用 Docker Compose。

```text
docker-compose
├─ web        Web 前台、后台、API
├─ worker     工作站服务执行 Worker
├─ db         数据库
├─ storage    文件存储挂载或外部对象存储适配
└─ proxy      可选，反向代理和 HTTPS
```

### 9.1 Docker 使用原则

- 容器内只放代码和运行环境。
- 数据库数据必须放在持久化卷或外部数据库。
- 上传文件不得只保存在容器内部。
- 镜像必须打版本号，不只使用 `latest`。
- Web 和 Worker 分开容器，便于独立更新和排查。

### 9.2 环境变量

环境变量用于：

- 数据库连接。
- Session 密钥。
- 文件存储配置。
- Worker 执行配置。
- 脚本运行参数。

禁止：

- 在前端暴露敏感环境变量。
- 将生产密钥提交到代码仓库。

## 10. 更新、备份与回滚

### 10.1 更新流程

```text
确认新版本镜像
  ↓
备份数据库和上传文件
  ↓
停止写入或进入维护窗口
  ↓
执行数据库迁移
  ↓
启动新版本 Web 和 Worker
  ↓
执行健康检查
  ↓
执行关键链路回归
  ↓
完成更新
```

### 10.2 备份策略

需要备份：

- 数据库。
- 上传附件。
- 导出文件，如有保留。
- 配置文件。

建议：

- 更新前必须手动或自动备份。
- 定期做数据库快照。
- 备份文件应加访问控制。

### 10.3 回滚策略

回滚要求：

- 保留上一个稳定版本镜像。
- 数据库迁移应有回滚脚本或兼容策略。
- Web 与 Worker 可回滚到同一版本。
- 回滚后执行登录、内容访问、工作站服务查询、大屏访问等关键链路检查。

### 10.4 为什么 Docker 不会妨碍更新

Docker 更新是否方便，取决于数据是否正确外置。

正确做法：

- 代码在镜像中。
- 数据库在持久化卷或外部数据库中。
- 上传文件在挂载目录或对象存储中。
- 版本用镜像标签管理。

这样更新时替换容器不会丢失数据，反而更容易回滚。

## 11. 安全设计

| 风险 | 设计措施 |
| --- | --- |
| 访客越权访问邀请内容 | 后端 API 强制鉴权 |
| 邀请用户查看他人提交记录 | 查询条件绑定当前用户 ID |
| 学生 App 密码泄露 | 不落库、不写日志、不导出 |
| 原始执行日志泄露 | 先脱敏再存储 |
| 作品附件被公开下载 | 下载接口鉴权，不暴露永久公开地址 |
| 留言刷屏 | 频率限制和内容长度限制 |
| 后台误删内容 | 危险操作二次确认，内容软删除 |
| 导出文件泄露 | 仅管理员导出，导出字段白名单 |

## 12. 性能与稳定性设计

- 工作站服务使用异步执行，避免请求长时间阻塞。
- 前端轮询服务状态，建议 2 秒一次。
- 任务执行设置超时时间。
- 后台列表使用分页查询。
- 大屏看板读取聚合数据，不直接扫访问明细。
- 访问明细定期聚合和清理。
- 执行日志按 180 天保留策略清理。

## 13. 后续可扩展点

- 工作站服务按用户或用户组授权。
- 管理员角色拆分为内容管理员、工作站管理员、审计管理员。
- 数据库任务表升级为消息队列。
- 文件存储从本地切换到对象存储。
- 导出从同步 CSV 扩展为异步任务。
- 大屏看板支持全屏模式和自动刷新配置。
- 日志自动提交脚本接入真实学校系统后增加专项联调用例。

## 14. 待确认项

- 最终技术栈，例如 Next.js 全栈、React + 后端服务、Vue + 后端服务等。
- 数据库类型。
- Cookie Session 的具体安全配置。
- Worker 与脚本之间的调用方式，例如命令行、内部模块、HTTP、本地队列。
- 访问明细短期保留天数。
- 数据库迁移工具。
- 文件上传大小限制和文件类型白名单。
